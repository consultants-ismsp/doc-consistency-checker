# 문서 정합성 검사기 — 개발 설계서

작성일: 2026-06-19
버전: v0.3 (LLM 공급자 교체 설계 반영)
대상: 직접 바이브코딩으로 개발하는 본인

---

## 0. 한 줄 정의

`.docx` 문서 세트를 넣으면, **문서 간 비교**와 **입력한 룰**에 따라
① 용어 불일치 ② 수치 불일치 ③ 판정·연계·종속관계 모순을
찾아 "어느 문서 어디가 어긋났는지"를 **HTML 대시보드**로 보여주는
**읽기 전용 범용 검사기**. 수정은 사람이 직접 한다.

---

## 1. 배경 / 목적

- ISMS-P 컨설팅 산출물(현황보고서·GAP·위험분석·이행계획 등)을 만들며
  문서 간 정합성을 수작업으로 대조하다 보니 놓침·모순이 발생한다.
- LLM에게 충족 여부를 판정시키면 정확도가 떨어지고(종속관계 무시 등) 못 믿는다.
  → **판정은 사람·코드가 하고, LLM은 추출/의미비교만** 시킨다.
- 프로젝트를 진행하며 자연스럽게 나온 QA 도구.
- 엔진은 도메인을 모르게 만들고, 룰·사전만 교체하면 타 도메인에도 쓴다(범용).

---

## 2. 설계 원칙 (가장 중요)

| 작업 | 담당 | 이유 |
|---|---|---|
| 숫자·연산·일치 비교 | **코드** | LLM은 정밀 연산에 약함(자릿수·합계 틀림) |
| 논리규칙·종속관계 강제 | **코드(룰엔진)** | LLM은 항목을 국소적으로만 봐 전역 제약 못 지킴 |
| 텍스트에서 값 추출 | **LLM(Solar)** | 비정형 문서 읽기는 LLM 강점 |
| 용어 의미 동일성 판단 | **LLM(Solar)** | "삭제=파기인가" 같은 의미 비교 |
| 최종 판정·표준 결정·수정 | **사람** | 도구는 적발만 |

추가 원칙:

- 도구는 **"적발"만** 한다. 수정·판정·표준선언은 하지 않는다(읽기 전용).
- **충족 여부(내용이 옳은가)는 안 본다.** 오직 **자기모순(정합성)** 만 본다.
- 엔진(범용) + 룰/사전(프로젝트별)을 분리해 범용성을 확보한다.
- 결과는 항상 **출처(문서·위치·원문)** 를 달아 사람이 바로 확인 가능하게 한다.
- **LLM 공급자(API)를 추상화**해, 코드 수정 없이 설정만으로 교체 가능하게 한다(§6.6).
  프로토타입은 Upstage Solar로 구현하되, 실무에선 고객사 승인 AI·보안회사 자체 AI로 교체한다.

---

## 3. 범위

**In (이번에 만든다)**
- 입력: `.docx` 다수
- 탐지: 용어 불일치 / 수치 불일치 / 판정·연계·종속관계 모순
- 출력: HTML 대시보드(정적 1파일)

**Out (안 한다 / 1차 제외)**
- 자동 수정(치환)
- 충족 여부 판정
- 취약점 진단·모의해킹·PoC (멘토 방침상 범위 밖)
- `.xlsx` / `.pdf` 입력 (1차 제외, 추후 확장 여지만)
- 이미지(캡처) 안의 텍스트 인식

---

## 4. 전체 아키텍처

```
[입력]                  [엔진(범용)]                          [출력]
docx 세트  ─┐
documents.yaml ─┼─► loader ─► extractor(LLM) ─► normalize ─► rule engine ─► report
(glossary.yaml)─┤   (파싱)     (항목 추출 JSON)    (공통모델)    (코드 검사)    (HTML
rules.yaml     ─┘                  │                                          대시보드)
                              LLM API (Solar)
```

**파이프라인 (데이터가 어떻게 흐르나)**

1. **load** — docx → 단락/표 + 위치 메타. 입력: 파일. 출력: `ParsedDoc`.
2. **extract** — `ParsedDoc` + 역할 → LLM(Solar)로 구조화 추출. 출력: `ExtractedDoc(JSON)`.
3. **normalize** — 여러 `ExtractedDoc` → 비교 가능한 공통 모델(`Model`).
4. **check** — `Model` + 룰/사전 → 위반 목록(`Finding[]`).
5. **report** — `Finding[]` → `dashboard.html`.

각 단계 산출물은 파일로 떨궈서(중간 JSON) 디버깅·재현이 되게 한다.

## 5. 입력 명세

### 5.1 문서 + 로더

- `.docx` N개.
- 각 문서에 **역할(role)** 지정 필수(자동 판별 안 함 — 신뢰성).

로더(`loader.py`)는 `python-docx`로 다음을 뽑는다.

| 추출 대상 | 보존할 위치 메타 |
|---|---|
| 단락 텍스트 | 단락 인덱스 |
| 표 셀 텍스트 | (표 번호, 행, 열) |
| 제목(Heading) | 절 구조 추정용 |

위치 표기(`section`)는 Heading 계층을 누적해 "3.2" 형태로 추정한다.
명시적 절 번호가 본문에 있으면 그걸 우선 사용한다.

```python
# loader.py 핵심 구조
class ParsedDoc:
    file: str
    role: str
    blocks: list[Block]      # 단락/표 셀을 순서대로

class Block:
    text: str
    kind: str                # "para" | "cell"
    section: str             # 추정 절번호 "3.2"
    para_index: int
    table_pos: tuple | None  # (표번호, 행, 열)
```

### 5.2 문서-역할 매핑 — `documents.yaml`

```yaml
documents:
  - file: 현황보고서.docx
    role: status_report
  - file: GAP분석.docx
    role: gap
  - file: 이행계획서.docx
    role: plan
  - file: 위험분석.docx
    role: risk
  - file: 자산대장.docx
    role: asset
```

### 5.3 용어 처리 — 2모드 (핵심 변경)

용어는 **사전 없이도 동작**하게 만든다. 사전은 선택이고, 쓰면 점점 똑똑해진다.

**모드 A — 비교 모드 (기본, 사전 불필요)**
- 문서들에서 같은 대상을 가리키는 표현이 **문서마다 다른지**를 본다.
- 예: 처리방침=「파기」, 내부관리계획=「삭제」 → "표현 갈림" 적발.
- 도구는 어느 게 표준인지 모른다. **둘 다 보여주고 사람이 표준 결정.**
- 발견된 변형 묶음은 사전 후보로 출력 → 점진적으로 사전이 쌓인다.

**모드 B — 사전 모드 (선택, 절대 기준 추가)**
- `glossary.yaml`이 있으면 "표준어 외 변형 사용"을 절대 기준으로 검사.
- 비교 모드의 사각지대(모든 문서가 똑같이 비표준어를 써서 차이가 안 나는 경우)를 잡는다.

> **사각지대 명심**: 모드 A만 쓰면 "전 문서가 똑같이 '삭제'로 통일" 된 경우 차이가
> 없어 못 잡는다. 그건 모드 B(사전)가 있어야 잡힌다. 그래서 둘은 병행한다.

```yaml
# glossary.yaml (모드 B, 선택)
terms:
  - standard: 파기
    variants: [삭제, 폐기, 없앰]
    except_context: [파일 삭제, 로그 삭제]   # 동음이의 예외
  - standard: 개인정보처리자
    variants: [정보처리자, 처리자]
```

### 5.4 룰 설정 — `rules.yaml`

```yaml
rules:
  # 수치 일치
  - id: count-match
    type: value_match
    key: 미흡건수
    must_equal_across: [status_report, gap, plan]

  # 검산
  - id: count-formula
    type: value_formula
    expr: "미흡건수 == 미흡 + 취약"

  # 연계(포함): GAP 미흡 항목 ID ⊆ 이행계획 대상 ID
  - id: gap-to-plan
    type: cross_reference
    source: {role: gap, field: 미흡항목ID}
    target: {role: plan, field: 이행과제대상ID}
    relation: subset

  # 종속(구조적, 계층 기반 — 전체 자동)
  - id: parent-child
    type: dependency
    mode: structural
    rule: parent_not_y_then_child_not_y   # 부모 N/P면 자식 Y 불가

  # 종속(의미적, 핵심만 수기)
  - id: risk-to-control
    type: dependency
    mode: semantic
    if:   {item: "1.2", verdict: N}
    then_not: {item: "1.3", verdict: Y}

  # 자산 정합
  - id: asset-subset
    type: subset
    source: {role: risk, field: 대표자산}
    target: {role: asset, field: 자산명}
```

룰 타입별 필수 필드:

| type | 필수 필드 | 의미 |
|---|---|---|
| `value_match` | key, must_equal_across | 키 값이 역할들 간 동일 |
| `value_formula` | expr | 키 값 간 산식 참 |
| `cross_reference` | source, target, relation | 집합 포함관계 |
| `dependency` | mode(+ structural rule / semantic if·then_not) | 판정 종속 |
| `subset` | source, target | 부분집합 |

## 6. 항목 추출 (LLM, Solar)

LLM은 **값 추출 + 용어 의미비교만** 한다. 비교·검산·판정은 코드가 한다.

### 6.1 역할별 추출 프로파일

| 역할 | 추출 대상 |
|---|---|
| `gap` | 통제항목ID, 판정값(Y/P/N), 미흡 사유 |
| `plan` | 이행과제명, 대상 통제항목ID, 우선순위 |
| `status_report` | 주요 수치(미흡건수·취약건수·보안수준 등) |
| `risk` | 대표자산명, 위험도 |
| `asset` | 자산명 목록 |
| (공통) | 용어 등장 위치(비교 모드용) |

### 6.2 Solar API 사용 (OpenAI 호환)

- Upstage Solar는 **OpenAI 호환 API**다(`base_url`만 다름). → openai 파이썬 SDK 그대로 사용.
- 추후 전환 대상: **OpenAI(GPT)·Gemini는 같은 구현**에 base_url만 교체(둘 다 OpenAI 호환 엔드포인트 제공). **Claude로 갈 땐 어댑터 1개만 추가.** (로컬은 안 씀, API 전용.)
- 모델: 한국어 추출·용어 비교에 Solar Pro 계열. 단순 추출 多·의미 비교 少로 단계별 분리 가능.
- **구조화 출력**: `response_format`(JSON) 또는 스키마 명시 프롬프트 + 파싱·검증.
- 파싱 실패 시 1~2회 재시도, 그래도 실패면 해당 문서 에러로 기록(중단 안 함).
- 문서가 길면 블록 단위로 청크해서 호출 후 병합.
- **값은 원문 그대로만**. 추론·계산·정규화(숫자 변환 등) 금지. 단위/표기도 원문 유지.

### 6.3 추출 프롬프트 템플릿 (예: gap)

```
당신은 문서에서 값만 그대로 추출하는 도구입니다. 판단·계산·요약하지 마세요.
아래 GAP 분석 문서 블록에서 다음을 JSON으로 추출하세요.
- 통제항목 판정: { key: 항목번호, value: Y|P|N }
- 각 값에는 출처(section, para_index, snippet) 포함
원문에 없는 값은 만들지 마세요. 애매하면 비워두세요.
[문서 블록들 + 위치 메타]
```

### 6.4 출력 JSON 스키마

```json
{
  "doc": "GAP분석.docx",
  "role": "gap",
  "verdicts": [
    { "key": "2.6.1", "value": "N",
      "source": {"section": "3.2", "para_index": 47, "snippet": "2.6.1 ... 미흡"} }
  ],
  "numbers": [
    { "key": "미흡건수", "value": 169,
      "source": {"section": "2.1", "para_index": 12, "snippet": "총 169건"} }
  ],
  "term_hits": [
    { "term": "삭제", "concept_hint": "데이터 파기",
      "source": {"section": "3.2", "para_index": 50, "snippet": "...삭제 절차..."} }
  ]
}
```

### 6.5 중간 산출물 검토

추출 JSON을 `out/extracted/*.json`으로 저장 → 검사 전에 사람이 오추출을 거를 수 있게.

### 6.6 LLM 공급자 추상화 (API 교체 가능)

LLM을 직접 부르지 않고 **공통 인터페이스(LLMProvider)** 뒤에 둔다.
추출·클러스터링 코드는 공급자가 누구인지 모른다 → 설정만 바꾸면 교체된다.

```python
# llm/base.py
class LLMProvider:
    def complete(self, system: str, user: str, json_schema: dict) -> dict:
        """구조화 JSON 응답을 돌려준다. 구현체마다 내부 호출 방식만 다름."""
        raise NotImplementedError

# llm/openai_compatible.py -> Solar/OpenAI(GPT)/Gemini 공용 (base_url만 다름, 기본=Solar)
# llm/anthropic.py         -> Claude 어댑터 (비OpenAI 호환, 전환 대상)
```

공급자 선택은 설정으로 한다.

```yaml
# settings.yaml
llm:
  provider: solar               # solar | openai | gemini | claude  (현재 solar, 추후 1개로 전환)
  base_url: https://api.upstage.ai/v1   # openai/gemini는 각 OpenAI 호환 엔드포인트로 교체
  model: solar-pro2             # 콘솔에서 현재 버전 확인
  api_key_env: UPSTAGE_API_KEY  # 키는 코드 아닌 환경변수로
```

extractor는 이렇게만 쓴다(공급자 무관).

```python
provider = make_provider(settings.llm)   # 팩토리
result = provider.complete(system, user, schema)
```

**왜 교체가 중요한가 (단순 편의가 아님)**
- 보안컨설팅 산출물엔 개인정보·민감정보가 들어 있어, 외부 공개 API로 전송하는 것
  자체가 보안 이슈다(ISMS-P 맥락에선 특히).
- 따라서 실무 적용 시엔 **고객사 내부망 AI나 보안회사 자체 AI**로 교체하는 게 정석.
- 발표 시 정직한 표현: "프로토타입은 Upstage Solar(국내 모델)로 구현, 실무에선
  고객사/자체 API로 교체 가능하도록 LLM 계층을 분리 설계." (안 쓴 API를 "썼다"고 하지 않는다.)

> 교체 시 바뀌는 건 `llm/` 구현 한 곳뿐. extractor·terms·rules·report는 손대지 않는다.
> 단, 모델마다 JSON 출력 습성·지시 따르는 방식이 달라 **전환 시 프롬프트 미세조정이 필요할 수 있다.**
> Solar→Claude 전환 시 더미 문서(T1~T7)로 추출 품질을 재검증할 것.

## 7. 용어 비교 엔진 상세

오탐이 제일 나기 쉬운 부분이라 단계를 나눈다.

1. **수집** — 모든 문서의 `term_hits` 모음.
2. **개념 클러스터링** — 같은 대상을 가리키는 표현 묶기(예: 파기/삭제/폐기).
   - LLM에게 "이 표현들이 같은 개념인가" 묶음 판단을 맡긴다(의미 비교는 LLM 강점).
   - 단, 자동 확정하지 않고 **후보**로만.
3. **불일치 판정(코드)**
   - 모드 A: 한 개념 클러스터가 문서마다 다른 표현이면 → Finding.
   - 모드 B: 사전 있으면 표준어 외 표현 → Finding(`except_context`는 제외).
4. **출력** — 어느 문서가 어떤 표현을 썼는지 표로. 표준 지정은 사람 몫.

오탐 방지:
- 동음이의는 `except_context`로 제외.
- 클러스터 신뢰도가 낮으면 "확인 필요"로 약하게 표시(심각도 low).

## 8. 룰 엔진 상세 (타입별 평가 로직)

```python
# value_match: 키 값이 지정 역할들에서 모두 같아야
vals = [m.number(role, key) for role in must_equal_across]
if len(set(vals)) > 1:
    finding("수치 불일치", expected="동일", actual=dict(zip(roles, vals)))

# value_formula: 추출값 대입 후 코드가 계산(LLM 아님)
if not eval_safe(expr, m.numbers):
    finding("검산 불일치", expr=expr, actual=...)

# cross_reference (subset): source ⊆ target
missing = set(source_ids) - set(target_ids)
if missing:
    finding("연계 누락", actual=list(missing))

# dependency / structural: 계층 순회
for parent, child in hierarchy_pairs(m.verdicts):
    if parent in ("N","P") and child == "Y":
        finding("구조적 종속 모순", parent=..., child=...)

# dependency / semantic: if -> then_not
if m.verdict(if.item) == if.verdict and m.verdict(then_not.item) == then_not.verdict:
    finding("의미적 종속 모순", ...)

# subset (자산): source ⊆ target, 차집합 적발
```

모든 위반은 `Finding`으로 통일(§13).

## 12. 처리 흐름 (의사코드)

```python
def run(config_dir):
    docs     = load_yaml(config_dir, "documents.yaml")
    rules    = load_yaml(config_dir, "rules.yaml")
    glossary = load_yaml_optional(config_dir, "glossary.yaml")  # 없어도 됨

    extracted = []
    for d in docs:
        parsed = loader.parse(d.file, d.role)
        try:
            items = extractor.extract(parsed, d.role)   # LLM(Solar)
        except ExtractError as e:
            log_error(d.file, e); continue
        save_json(f"out/extracted/{d.role}.json", items)
        extracted.append(items)

    model = normalize.build(extracted)

    findings = []
    findings += terms.check(model, glossary)   # 모드 A(+B)
    for r in rules:
        findings += engine.run(r, model)

    report.render_html(findings, model, "out/dashboard.html")
    print_summary(findings)
```

## 21. 앱 패키징 / 크로스플랫폼 (맥 → 윈도우 .exe)

목표: 지금은 맥 앱, 추후 윈도우 `.exe`로도 배포할 수 있게.

### 21.1 핵심 전략 — 코어 / UI 셸 분리

- **코어(순수 파이썬)**: loader·extractor·terms·rules·report·llm.
  OS와 무관 → Mac/Windows에서 그대로 동작. 이식 비용 0.
- **UI 셸(얇게)**: "파일 선택 → 코어 `run()` 호출 → `dashboard.html` 열기".
  OS 의존(파일 다이얼로그·패키징)은 이 한 겹에만 둔다.
- LLM 공급자 추상화(§6.6)와 같은 원리: **갈아끼우는 부분을 가장자리로** 몬다.

> 결과물이 HTML이라 UI는 두꺼울 필요가 없다. 무거운 화면은 브라우저(대시보드)가 맡는다.

### 21.2 프레임워크 비교 (.exe 이식 관점)

| 프레임워크 | 크로스플랫폼 | .app/.exe 패키징 | 의존성 | 룩 | 추천 |
|---|---|---|---|---|---|
| **Tkinter** | O (내장) | PyInstaller로 둘 다 쉬움 | 0 | 투박 | ★ 1순위 |
| **PySide6/Qt** | O | PyInstaller 표준, 용량 큼 | 큼 | 좋음 | 2순위 |
| **Streamlit** | O(로컬 웹서버) | 단일 `.exe` 배포 약함 | 큼 | 웹 | 배포 목표면 비추 |

### 21.3 권장

- **1순위: Tkinter 얇은 셸 + PyInstaller.**
  내장이라 의존성 0, `.app`·`.exe` 둘 다 가장 깔끔. UI는 파일 선택+실행 버튼+
  대시보드 열기뿐이라 투박해도 충분. **`.exe` 이식이 우선이면 이게 정답.**
- **2순위: PySide6** — 룩이 중요할 때.
- **Streamlit** — 개발은 빠르나 더블클릭 배포가 약함. "파이썬 깔린 환경 실행"만이면 OK.

### 21.4 패키징 방법

- **Mac**: `pyinstaller --windowed app.py` → `.app`. (또는 Briefcase)
- **Windows**: 같은 코드, 윈도우에서 `pyinstaller` → `.exe`.
- **주의**: PyInstaller는 **빌드한 OS의 바이너리만** 만든다(크로스 빌드 불가).
  → `.exe`는 윈도우에서 빌드해야 한다. 코드는 그대로라 빌드만 다시 돌리면 됨.

### 21.5 플랫폼 의존 주의점 (처음부터 지킬 것)

- 파일 경로: `pathlib` 사용, 슬래시 하드코딩 금지.
- 인코딩: docx 읽기·HTML 출력 **UTF-8 고정**(한글 깨짐 방지).
- 대시보드 열기: `webbrowser.open()` (OS 무관).
- 임시 폴더: `tempfile` 사용.
- API 키: 환경변수(`api_key_env`)로, 코드·설정에 하드코딩 금지. OS별 설정법만 안내.

## 22. 개발 표준 (코딩 시 준수)

### 22.1 객체지향 설계

- **모듈 = 단일 책임 클래스.** loader/extractor/normalize/terms/report 각각 한 가지만.
- **교체 지점은 추상클래스로 분리** — 이미 `LLMProvider`. 같은 방식으로:
  - `BaseRule.check(model) -> list[Finding]` → 룰 타입별 서브클래스
    (`ValueMatchRule`, `CrossRefRule`, `DependencyRule`, `SubsetRule`).
  - `BaseReporter.render(findings)` → `HtmlReporter` 등.
- **의존성 주입** — `Pipeline`이 provider·rules·reporter를 생성자로 주입받는다.
  전역 변수·전역 상태 금지(테스트·교체가 쉬워짐).
- **데이터는 dataclass** — `Source`, `ExtractedItem`, `Finding`(schema.py).
- 룰을 데이터(yaml)로 받되, 실행은 클래스가 한다(설정 → 룰 객체 매핑).

```python
# rules/base.py
class BaseRule:
    def __init__(self, spec: dict): self.spec = spec
    def check(self, model) -> list[Finding]: raise NotImplementedError

# 설정의 type 문자열 → 룰 클래스 매핑
RULE_MAP = {"value_match": ValueMatchRule, "dependency": DependencyRule, ...}
```

### 22.2 동적 자원 경로 (PyInstaller 함정 대응)

패키징하면 `prompts/` 같은 번들 자원의 경로가 바뀐다(`sys._MEIPASS`).
경로를 하드코딩하면 개발 땐 되고 배포 후 깨진다. **자원 경로 헬퍼를 반드시 쓴다.**

```python
# core/paths.py
import sys, os
from pathlib import Path

def resource_path(rel: str) -> Path:
    # PyInstaller로 묶이면 _MEIPASS, 아니면 소스 기준
    base = getattr(sys, "_MEIPASS", Path(__file__).resolve().parent.parent)
    return Path(base) / rel

# 사용: prompt = resource_path("core/prompts/extract_gap.txt").read_text(encoding="utf-8")
```

### 22.3 읽기 자원 vs 쓰기 데이터 분리

- **읽기 전용 번들 자원**(prompts 등) → `resource_path()`로 접근. 번들 안엔 못 쓴다.
- **쓰기 데이터**(out/, 추출 JSON, dashboard.html) → 번들이 아닌
  사용자 작업 폴더/홈 밑에 둔다. 패키징된 앱 내부에 쓰려 하면 권한 오류.

### 22.4 OS 파일경로 호환성

- 항상 `pathlib.Path`. 문자열 `+`·슬래시 하드코딩 금지.
- 파일 입출력은 **`encoding="utf-8"` 명시**(한글 깨짐·OS 기본 인코딩 차이 방지).
- 한글·공백 경로는 `Path`가 처리하지만, 셸 호출 시엔 인용 주의.
- 출력 폴더는 `Path.mkdir(parents=True, exist_ok=True)`로 보장.
- 임시 파일은 `tempfile`, 줄바꿈·구분자는 `os` 모듈에 위임.

### 22.5 그 외

- 예외는 단계별로 잡아 메시지에 **문서·위치**를 남긴다(어디서 터졌는지).
- 로깅은 `logging` 모듈(print 산발 금지), 레벨로 디버그/운영 구분.
- 설정·키는 코드 밖(yaml·환경변수). 비밀키 커밋 금지.
