# Claude Code 개발 프롬프트 — 문서 정합성 검사기

> 사용법: 아래 전체를 Claude Code에 붙여넣고 시작. 같은 폴더의
> `문서정합성검사기_설계서.md`를 함께 열어 두면 세부 스펙을 참조할 수 있다.

---

## 0. 역할과 목표

너는 파이썬 개발자다. **`.docx` 문서 세트의 정합성을 검사하는 맥/윈도우 데스크톱 앱**을
만든다. 사용자가 만든 컨설팅 산출물 여러 개를 넣으면, 문서들 사이의
**용어 불일치 · 수치 불일치 · 판정/연계/종속관계 모순**을 찾아
**HTML 대시보드**로 "어느 문서 어디가 어긋났는지"만 보여준다. **수정은 하지 않는다.**

한 번에 다 짜지 말고 아래 "개발 순서"대로 단계별로 만들고, 각 단계마다 테스트로 검증하라.

---

## 1. 절대 원칙 (어기지 말 것)

1. **역할 분담을 지켜라.**
   - 숫자·연산·일치 비교 → **코드**가 한다.
   - 논리규칙·종속관계 강제 → **코드(룰엔진)**가 한다.
   - 텍스트에서 값 추출, 용어 의미 비교 → **LLM**이 한다.
   - **LLM에게 판정(Y/N)·계산·비교를 시키지 마라.** LLM은 추출과 의미 클러스터링만.
2. 도구는 **적발만** 한다. 자동 수정·충족 판정·표준 선언을 하지 않는다.
3. 모든 결과(Finding)에는 **출처(문서명·절·단락·원문 스니펫)**를 붙인다.
4. 엔진은 도메인을 모른다. 룰·용어사전·문서역할은 전부 **외부 설정(yaml)**으로 받는다.
5. **코어(순수 파이썬) / UI 셸**을 분리한다. OS·프레임워크 의존은 UI 셸에만.
6. **LLM 공급자를 추상화**한다. 현재 Solar, 추후 Gemini/Claude/GPT 중 1개로 설정만 바꿔 전환 가능해야 한다(로컬은 안 씀).

---

## 2. 기술 스택

- 언어: Python 3.11+
- docx 파싱: `python-docx`
- 설정: `PyYAML`
- LLM: Upstage Solar API (OpenAI 호환, §6 추상화 뒤에 둠 / 추후 Gemini·Claude·GPT 중 1개로 전환)
- UI 셸: **Tkinter**(내장) — 파일 선택 + 실행 버튼 + 대시보드 열기
- 패키징: PyInstaller (`.app` / 윈도우에서 `.exe`)
- 외부 의존 최소화. UI는 얇게, 무거운 화면은 브라우저(HTML 대시보드)가 맡는다.

## 4. 데이터 모델 (core/schema.py, dataclass)

```python
@dataclass
class Source:
    doc: str; section: str; para_index: int; snippet: str

@dataclass
class ExtractedItem:
    doc: str; role: str; field: str       # verdict|number|term
    key: str | None; value: str | int; source: Source

@dataclass
class Finding:
    rule_id: str; type: str               # term|value|cross_ref|dependency|subset
    severity: str                         # high|medium|low
    docs: list[str]; locations: list[Source]
    expected: str; actual: str; message: str
```

## 5. 설정 파일 스펙

```yaml
# settings.yaml
llm:
  provider: solar             # solar | openai | gemini | claude (현재 solar, 추후 1개로 전환)
  base_url: https://api.upstage.ai/v1   # openai/gemini는 각 OpenAI 호환 엔드포인트로 교체
  model: solar-pro2           # 콘솔에서 현재 버전 확인
  api_key_env: UPSTAGE_API_KEY

# documents.yaml
documents:
  - {file: 현황보고서.docx, role: status_report}
  - {file: GAP분석.docx,   role: gap}
  - {file: 이행계획서.docx, role: plan}
  - {file: 위험분석.docx,   role: risk}
  - {file: 자산대장.docx,   role: asset}

# rules.yaml
rules:
  - {id: count-match, type: value_match, key: 미흡건수, must_equal_across: [status_report, gap, plan]}
  - {id: count-formula, type: value_formula, expr: "미흡건수 == 미흡 + 취약"}
  - {id: gap-to-plan, type: cross_reference,
     source: {role: gap, field: 미흡항목ID},
     target: {role: plan, field: 이행과제대상ID}, relation: subset}
  - {id: parent-child, type: dependency, mode: structural, rule: parent_not_y_then_child_not_y}
  - {id: risk-to-control, type: dependency, mode: semantic,
     if: {item: "1.2", verdict: N}, then_not: {item: "1.3", verdict: Y}}
  - {id: asset-subset, type: subset,
     source: {role: risk, field: 대표자산}, target: {role: asset, field: 자산명}}

# glossary.yaml (선택, 모드 B)
terms:
  - {standard: 파기, variants: [삭제, 폐기], except_context: [파일 삭제, 로그 삭제]}
```

## 6. LLM 추상화 (core/llm)

```python
# base.py
class LLMProvider:
    def complete(self, system: str, user: str, json_schema: dict) -> dict: ...
```

- `openai_compatible.py`: Solar/OpenAI/Gemini 공용 구현. openai SDK에 `base_url`만 다르게.
  구조화 JSON(`response_format`) 강제, 파싱 실패 시 1~2회 재시도.
- `anthropic.py`: Claude로 전환할 때만 추가하는 어댑터(비OpenAI 호환). 지금은 스텁.
- 모델 전환 시 프롬프트 미세조정이 필요할 수 있다. Solar→Claude 전환 시 더미 문서(T1~T7)로 추출 품질 재검증.
- `factory.py`: `settings.llm.provider` 보고 공급자 생성.
- extractor·terms는 `LLMProvider`만 알고 구현체는 모른다.
- **값은 원문 그대로 추출**. 추론·계산·정규화 금지.

## 7. 검사 기능 명세

### 용어 (terms.py) — 2모드
- **모드 A(기본, 사전 불필요)**: 같은 개념을 문서마다 다른 표현으로 쓰면 적발.
  개념 묶기는 LLM(후보만), 불일치 판정은 코드. 표준은 사람이 정함(심각도 low).
- **모드 B(사전 있으면)**: 표준어 외 변형 사용 적발. `except_context` 제외(심각도 medium).
- 사각지대: 모든 문서가 똑같이 비표준어를 쓰면 모드 A는 못 잡음 → 모드 B로만.

### 룰 (rules/) — 타입별 클래스, BaseRule.check(model)->Finding[]
- `value_match`: 키 값이 역할들에서 모두 동일한가.
- `value_formula`: 추출값 대입해 **코드가** 산식 평가(LLM 아님).
- `cross_reference(subset)`: source 집합 ⊆ target 집합, 차집합이 누락.
- `dependency/structural`: 계층에서 부모가 N/P인데 자식 Y면 모순(전체 자동).
- `dependency/semantic`: if 항목판정 성립 & then_not 성립이면 모순(핵심만 수기).
- `subset`: 자산 정합 등 부분집합.

## 8. 출력 (core/report.py) — HTML 대시보드

- 정적 HTML 1파일, 인라인 CSS/JS, 외부 의존 없음.
- 구성: 헤더(일시·대상문서·총건수) / 요약 카드(유형·심각도별) / 필터(JS) /
  Finding 상세 카드(유형·심각도 배지, 메시지, 문서+위치, 원문 스니펫, 기대 vs 실제).
- 데이터는 HTML 안에 JSON 인라인 → JS가 렌더·필터.
- `out/dashboard.html`에 저장 후 `webbrowser.open()`.

## 9. 코딩 표준 (반드시 지킴)

- **객체지향**: 모듈=단일책임 클래스. 교체지점은 추상클래스(`LLMProvider`,`BaseRule`,`BaseReporter`).
  `Pipeline`이 의존성을 주입받는다(전역 상태 금지).
- **동적 자원 경로**: 번들 자원(prompts 등)은 `core/paths.py`의 `resource_path()`로 접근.
  PyInstaller `_MEIPASS` 대응. 경로 하드코딩 금지.
  ```python
  def resource_path(rel):
      base = getattr(sys, "_MEIPASS", Path(__file__).resolve().parent.parent)
      return Path(base) / rel
  ```
- **읽기 자원 / 쓰기 데이터 분리**: 출력(out/)은 번들 밖 작업 폴더에. 번들 내부에 쓰지 마라.
- **OS 경로 호환**: 항상 `pathlib.Path`, 슬래시 하드코딩 금지.
  파일 입출력 `encoding="utf-8"` 명시. 출력 폴더 `mkdir(parents=True, exist_ok=True)`.
- 예외는 단계별로 잡고 메시지에 문서·위치를 남긴다. 로깅은 `logging`(print 남발 금지).
- 키·설정은 코드 밖(yaml·환경변수). 비밀키 커밋 금지.
- 주석은 짧고 자연스럽게. 과한 추상화·불필요한 설계는 피한다.

## 10. 개발 순서 (단계별로, 각 단계 끝에 테스트)

각 단계는 `tests/fixtures`에 모순을 심은 docx로 검증한다.

| 단계 | 만들 것 | 완료 기준 |
|---|---|---|
| 1 | schema.py, paths.py, loader.py | docx에서 블록+위치 추출됨 |
| 2 | terms.py 모드 A | 사전 없이 용어 갈림 탐지(T5) |
| 3 | llm 추상화 + extractor + prompts | 역할별 추출 JSON 생성, 중간 JSON 저장 |
| 4 | rules: value_match/value_formula | 수치 불일치·검산(T1,T2) |
| 5 | rules: cross_ref/dependency/subset | 연계·종속(T3,T4) |
| 6 | report.py | HTML 대시보드 생성·열기 |
| 7 | glossary 모드 B | 사전 위반(T6) |
| 8 | pipeline.py + app/main.py(Tkinter) | 파일 선택→실행→대시보드 |
| 9 | build 스크립트 | `.app` 생성, 윈도우 빌드 절차 문서화 |

테스트 케이스:
- T1 미흡건수 169 vs 168 → 수치 불일치
- T2 169 ≠ 140+28 → 검산 실패
- T3 GAP 2.6.1 미흡, 이행계획에 없음 → 연계 누락
- T4 부모 2.6=P, 자식 2.6.1=Y → 구조적 종속
- T5 문서마다 파기/삭제 혼용 → 용어 갈림(low)
- T6 전 문서 "삭제" + 사전 표준=파기 → 사전 위반(medium)
- T7 정합 문서 → 위반 0 (거짓양성 점검)

## 11. 작업 방식

- 한 단계씩 만들고, 테스트가 통과해야 다음으로 간다.
- 추출 스키마나 룰이 실제 문서와 안 맞으면 멈추고 **실제 산출물 샘플을 요청**하라.
- 의미적 종속 룰의 구체 항목쌍은 미정이니, 일단 구조적 종속만 전체 적용하고
  의미적은 예시 1개로 골격만 만든 뒤 확정 요청하라.
- 모르는 건 추측하지 말고 질문한다. 특히 문서 역할·필드명은 실제와 맞춰야 한다.
```
