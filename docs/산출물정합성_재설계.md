# 산출물 정합성 검사기 재설계 (v2 — 컨설팅 산출물 중심)

> 배경: 루키즈 생성형AI 보안컨설팅 프로젝트에서 **수행사(컨설턴트)가 만드는 산출물**의
> 용어·수치가 일관되지 않는 문제를 적발하는 게 원래 취지다. 도구의 1차 대상을
> "임의 문서 비교"에서 **수행사 산출물 세트(docx + xlsx)의 내부 정합성**으로 재정의한다.
>
> 변하지 않는 원칙(CLAUDE.md): 도구는 **적발만**(자동수정·충족판정·표준선언 금지),
> 완전 클라이언트사이드, 모든 Finding에 출처 부착, 숫자/논리=코드·텍스트추출=LLM.

---

## 0. 핵심 교정 (이번 설계의 출발점)

**RFP를 표준어 기준(anchor)으로 삼으면 안 된다.**
RFP는 발주처(고객)가 쓰는데, 그들은 보안 전문용어·체계를 모르기 때문에 컨설팅을 맡긴다.
즉 RFP는 **비전문 용어가 정상적으로 섞이는 입력 문서**다.

→ **표준어의 출처는 문서가 아니라 외부 표준용어집(`lib/config.ts`의 glossary)**,
  즉 컨설턴트의 도메인 지식이다. 산출물은 이 표준에 맞춰야 한다.

---

## 1. 문서 역할 분류 (term_authority 도입)

각 역할에 **용어 권위(term_authority)** 속성을 붙인다. 정합성 비교 대상/방향을 이 속성이 결정한다.

| 역할 | 분류 | term_authority | 주 형식 | 의미 |
|---|---|---|---|---|
| (외부) 표준용어집 glossary | 기준 | `authority` | config | 표준어를 **정의**한다. 문서가 아님. |
| RFP(제안요청서) | 입력(발주처) | `none` | docx | 비전문 용어 정상. **용어 정합 대상에서 제외.** |
| 수행계획서 | 산출물(수행사) | `subject` | docx | 표준 준수 대상. 결함은 여기서 적발. |
| 현황보고서 / GAP분석 / 이행계획서 / 위험분석 / 자산대장 / 수준평가보고서 | 산출물(수행사) | `subject` | docx·xlsx | 〃 |
| 일반문서(general) | (기존 호환) | `subject` | docx | 기존 동작 유지(기본값). |

- `authority`: 표준어를 정의. 현재는 config glossary. (팀이 "용어정의서"를 산출물로 만들면 그 역할도 authority 후보.)
- `subject`: 표준 준수 + 산출물끼리 일치해야 함. **결함 리포트의 대상.**
- `none`: 용어 정합 비교에서 빠짐. 정보성/추적 용도로만 쓰임.

## 2. 용어 검사 재설계

표준 우선순위: **① 외부 glossary(authority) → ② 산출물 간 일치.**

### Mode B (glossary 준수) — 거의 그대로, 강화만
- 기존: 표준어 외 변형 사용 적발(medium). `except_context` 제외.
- 변경: 검사 대상을 **`subject` 역할로 한정**(RFP의 비전문 용어를 변형으로 적발하지 않음).
- 이게 **주력 검사**가 된다(표준이 명확하므로).

### Mode A (문서 간 표현 갈림) — 역할 필터 추가
- 기존: 같은 개념을 문서마다 다르게 쓰면 적발(low, "표준은 사람이 결정").
- 변경 1: **`subject` 역할 문서끼리만** 비교. RFP(`none`)는 빼서 거짓양성 제거.
- 변경 2: glossary에 해당 개념의 표준어가 있으면 메시지에 **그 표준어를 권고**로 덧붙임
  (선언이 아니라 권고 — "적발만" 원칙 유지).

### RFP의 처리 — 제외(기본), 매핑은 정보성 옵션
- 기본: RFP는 용어 정합에서 **제외**.
- 선택 옵션(후속): "RFP는 「X」, 산출물 표준어는 「Y」" 같은 **단방향 매핑 정보**만 표시
  (불일치 아님 / finding 아님 / 또는 info 등급). 발주처 용어 ↔ 표준어 대응표 용도.

## 3. xlsx 로더 (설계만, 구현은 승인 후 다음 단계)

보안컨설팅 산출물의 상당수가 Excel(자산대장·위험분석표·GAP표·점검표·이행계획).
지금은 `word/document.xml`만 읽어 **이들을 아예 못 본다**.

- 파싱: jszip로 `xl/workbook.xml`(시트명) + `xl/sharedStrings.xml` + `xl/worksheets/sheet*.xml`.
- 셀 값: shared string / inline string / 숫자(`<v>`). 수식은 캐시값(`<v>`) 사용.
- **매핑(기존 `Block` 구조 재사용)**: 시트명 → `section`, (행,열) → `table_pos = [sheetNo, row, col]`, `kind="cell"`.
- 결과가 docx와 동일한 `Block[]` 이므로 **normalize / terms / rules / report 무수정**.
- 표 기반 룰(`asset-subset`, `gap-to-plan`)이 비로소 실제 xlsx 표에 동작 → 구조 규칙의 가치가 살아남.
- 완전 클라이언트사이드 유지, **새 의존성 0**(jszip 재사용).

## 4. 코드 영향 범위

| 파일 | 변경 | parity 영향 |
|---|---|---|
| `lib/config.ts` | 역할 메타(분류 + term_authority) 테이블 추가 | 없음 |
| `lib/prompts.ts` | ROLE_LABELS에 RFP·수준평가보고서 추가, 역할→authority 매핑 | 없음 |
| `lib/terms.ts` | Mode A/B에 역할(term_authority) 필터 인자 추가 | **주의(아래)** |
| `lib/loader.ts` | xlsx 분기 추가(다음 단계) | 없음(추출 단계) |
| `app/page.tsx` | 파일별 역할 선택지 확장, 파일명 기반 역할 추론 | 없음 |
| 엔진(normalize·rules·report)·결정적 비교 | **무수정** | 유지 |

### parity 무결성 — 가산적(additive)으로 보존
CLAUDE.md 불변식: "Python = 정답, 동작 바꾸면 parity 먼저 통과."
이 재설계는 **기존 동작을 깨지 않게 가산적으로** 넣는다:

- 역할 필터는 `term_authority`가 `none`인 hit만 제외한다.
- 기존 parity 픽스처는 전부 `general`(= `subject`)이라 **제외 대상이 0개** → 동작 동일 → **37/37 그대로 통과**.
- RFP 제외/단방향 매핑은 RFP 역할이 태깅된 새 입력에서만 발동(opt-in).

→ Python 정답 계약을 유지하면서 컨설팅 맥락 동작을 **추가**한다. 별도 spec 분기 불필요.

## 6. 확정된 결정 (2026-06-26)

- **D1. RFP 처리 = 완전 제외.** RFP(`reference` 역할)는 용어 정합(Mode A/B)뿐 아니라
  **자동 수치 비교(autoValueConsistency)에서도 제외**한다. 발주처 입력은 비전문 용어·다른
  맥락이 정상이므로 정합성 적발 대상이 아니다. (단방향 매핑 표시는 하지 않음.)
- **D2. 표준어 출처 = config glossary만.** 어떤 문서 역할도 `authority`로 두지 않는다.
- **D3. parity = 가산적 보존.** reference 역할이 0개인 기존 픽스처는 동작 불변 → 40 pass(기존 37 + RFP 제외 3).

## 9. 룰베이스 전면 재작성 (실제 산출물 0626 기준, 2026-06-26)

기존 룰셋(`RULES_DEFAULT`/`RULES_ASSESSMENT`/`RULES_ISMS_DEMO`)은 옛 가상 문서 기준이라 **전부 제거**하고,
실제 산출물 9종(`_inputs/`)을 인스펙터(`scripts/inspect-inputs.ts`)로 분석해 새 룰셋을 처음부터 작성.

**역할 재정의(prompts.ts)**: `plan`(수행계획서) · `level_eval`(수준평가보고서) · `risk_eval`(위험평가
결과보고서+xlsx) · `task_def`(이행과제정의서) · `asset`(자산관리대장) · `checklist`(체크리스트·법령매핑)
\+ `general`/`rfp`. 판정은 **Y/P/N 문자로 추출**(실문서 척도 양호=Y/미흡=P/취약=N이 엔진 한글매핑과 충돌하므로).
formula·value_match용 수치는 **공백 없는 정규 key**로 추출(예: 노출위험건수, 전체점검항목수).

**새 룰셋 `RULES_BRIDGEX`(config.ts)** — 엔진은 그대로, 룰 데이터만 교체:
- 검산(value_formula): `노출위험건수==높음+보통+낮음`, `미부분이행건수==부분이행+미이행`, `전체점검항목수==적용+제외`
- 문서간 일치(value_match): 전체점검항목수[plan·level_eval], 노출위험건수·미부분이행건수[risk_eval docx↔xlsx]
- 연계/부분집합(cross_reference·subset): 취약(N)항목⊆이행과제, 이행과제⊆체크리스트 통제번호, 위험평가 대표자산⊆자산대장
- 종속(dependency structural): 부모 N/P인데 자식 Y → 모순(계층 있을 때)
- 용어 glossary: `애플리케이션`(표준)/`어플리케이션`(변형) 추가

**검증**: tsc OK / `npm run parity` 38 pass(엔진 단위테스트는 합성 데이터라 유지, 실데이터는 결정적
용어엔진 재현으로 견고화) / build OK / `buildSampleResult()` 8건 적발(검산·일치·연계·부분집합·종속·용어 전부 시연, skip 0).
- 비고: 전체 파이프라인(추출+검사)은 브라우저+OpenRouter 키로만 실행 가능 → 실제 9파일은 `npm run dev`에서 확인 필요.

## 9-1. 문서 흐름 기반 추적 룰 보강 (2026-06-26)

산출물은 **법령매핑(checklist=통제항목 마스터) → 수행계획 → 자산 → 수준평가 → 위험평가 → 이행과제**
순으로 흐른다. 통제항목을 참조하는 산출물은 모두 마스터의 부분집합이어야 한다는 흐름에 맞춰 추적 룰 추가:

- **수준평가-마스터정의-누락**: 수준평가 평가항목 ⊆ 법령매핑 통제번호. (실데이터에서 `2.1.4`가 마스터에 없음을 적발)
- **이행과제-평가항목-추적**: 이행과제 대상 ⊆ 수준평가 평가항목(평가 안 된 항목을 가리키면 적발).
- 지원: `lib/pipeline.ts`의 `withDerivedFields()` — 수준평가 판정(verdict) 키를 `평가통제항목ID` 필드로
  **코드가 파생**(LLM 재추출 없음). run·runChecksOnly 양쪽 적용.
- 근거: `scripts/inspect-controls.ts`로 실데이터 포함관계 확인(checklist 101 / level 102(2.1.4 외부) / risk 72 ⊆ / task 16 ⊆ / task ⊆ level).
- 검증: tsc OK / parity 38 / build OK / 샘플 9건(high7·med1·low1, skip0) — 새 룰 `수준평가-마스터정의-누락` 적발 시연.

## 9-2. 공식 표준 용어 LLM 의미비교 (2026-06-26)

공식 안내서 2종 PDF(`_inputs/`)를 근거로 **표준 용어**를 도입. 통제번호 마스터는 하지 않음(사용자 결정).
- **표준 어휘 추출(오프라인 1회)**: 금융보안원 점검항목 안내서 PDF에서 통제항목명+분야명 **119개**를
  추출해 `config.ts`의 `STANDARD_TERMS`에 박음. (`scripts/extract-standard-terms.ts`, devDep `pdf-parse`. 런타임은 PDF 미사용.)
- **LLM 의미비교(런타임, 토글)**: 산출물이 쓴 용어를 `STANDARD_TERMS`와 **LLM이 의미비교**해, 의미는 같은데
  표기/띄어쓰기/표현이 다른 비표준 용어를 적발(예: '사용자계정관리'→표준 '사용자 계정 관리'). role-split 준수
  (의미비교=LLM, 적발=코드). 호출은 전체 distinct 용어 **1회**.
  - `prompts.ts` STANDARD_AUDIT, `extractor.auditStandardTerms()`, `pipeline.auditStandardTerms()`(reference 역할 제외),
    finding `standard-term:{표준어}`(medium). UI 토글 "공식 표준용어 점검(LLM)"(기본 ON), reference(RFP) 제외.
- 비고: 두 PDF에 용어정의표 없음 → 통제항목명/분야명을 표준 어휘로 사용. 텍스트 레이어 정상(OCR 불필요).
  PDF 추출 스크립트는 dev 전용(`pdf-parse` + `pdf-parse.d.ts` 앰비언트 선언).

## 9-3. 모드B 토글 제거 + 오타 점검 (2026-06-26)

- **용어사전(모드 B) 토글 제거**: glossary(애플리케이션·파기·개인정보처리자)는 무료·결정적이라 **항상 적용**으로
  바꾸고 UI 토글만 제거(`glossaryMode` 삭제, `glossary = GLOSSARY_DEFAULT` 고정).
- **오타 점검(LLM, 옵션)**: 추출과 **같은 패스**에서 명백한 오타/맞춤법 오류도 함께 보고(추가 호출 없음).
  `prompts.TYPO_INSTRUCTION`을 추출 프롬프트에 덧붙이고(토글 ON일 때만), `extractor`가 `typos`를 field="typo"
  항목으로 수집 → `pipeline.typoFindings()`가 finding(type **"typo"**, severity low, reference 역할 제외)으로 변환.
  새 유형 "오타"를 schema/report/Dashboard 필터·카드에 추가. UI 토글 "오타 점검(LLM)" 기본 OFF(노이즈·토큰 고려).
- 검증: tsc OK / parity 38 / build OK / 샘플 11건(오타 1·표준용어 1 합성 포함).
